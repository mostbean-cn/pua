# PUA L2 · 系统排查

这是重复失败或明显卡壳时的模式。

L2 的重点不是“更努力”，而是**停止凭感觉试错，改为系统化排查**。

> 不要再围着一个猜测转圈。读信号、补上下文、列假设、做排除，拿事实推进。

## 适用场景

- 同一问题已经失败 2 次以上
- 报错复杂，不能靠局部直觉解决
- 你怀疑自己开始原地打转
- 需要给出更扎实的排查路径，而不是继续微调

## 行为要求

1. **逐字核对失败信号**
   - 报错原文、返回码、日志、堆栈、异常位置，要按字面读。
   - 不要把“我以为它在说什么”当成“它真的在说什么”。

2. **读取原始上下文**
   - 读相关代码、配置或调用链，而不是只盯一行报错。
   - 至少补齐失败点附近的必要上下文，避免断章取义。

3. **搜索资料和已有线索**
   - 搜仓库内已有实现、历史模式、配置线索。
   - 需要时查官方文档、错误关键词、接口约束。
   - 没搜就下结论，属于偷懒。

4. **列出多个假设并逐个验证**
   - 每个假设都要对应验证动作。
   - 每次排除一个假设，都要缩小问题边界。
   - 不要只保留一个自己最喜欢的解释。

5. **前一条路失败，下一轮必须换本质不同的方案**
   - 换参数、换变量名、换一句措辞，不叫换方案。
   - 本质不同，指的是观察角度、验证手段、切入层级或实现路径发生变化。

6. **输出必须落到可执行的下一步**
   - 说明已经确认了什么
   - 已排除了什么
   - 下一步试什么，为什么试

7. **顺手检查直接相关问题**
   - 如果发现根因属于某一类模式，而不是单点偶发，至少指出同类风险范围。

## 三条红线

- **没有证据，不要归因。**
- **没有新信息，不要机械重试。**
- **没有换思路，不要声称已经穷尽。**

## L2 标准流程

1. **读失败信号**
2. **读上下文**
3. **搜资料 / 搜仓库 / 搜文档**
4. **列 2-3 个假设**
5. **逐个验证并排除**
6. **切换到本质不同的方案**
7. **汇总事实、排除项和下一步**

## “本质不同方案”的判定标准

以下至少满足一项，才算真的换了方案：

- 观察角度变了
- 验证手段变了
- 切入层级变了
- 实现路径变了
- 假设前提变了

只换参数、换名字、换一点局部写法，不算。

## 抗合理化清单

遇到下面这些念头，要立刻纠偏：

- **“可能是环境问题”** → 先验证环境，再说。
- **“应该不支持”** → 先搜文档和已有实现，再说。
- **“我都试过了”** → 把试过的方案列出来；少于 3 类，多半没穷尽。
- **“要不让用户手动处理”** → 先确认自己是不是还没完成系统排查。
- **“继续微调一次试试”** → 如果思路没变，通常是在原地打转。

## 常见陷阱

- **假装换了方案**：实际只是换参数或换个函数名。
- **只报现象，不报排除项**：看起来做了很多，实际上边界没有缩小。
- **嘴上说闭环，手里没证据**：结论和动作脱节。
- **问题已显露模式感，却仍只盯单点**：缺少同类风险意识。

## L2 输出格式要求

在阶段性汇报里，尽量包含：

- **已验证事实**：当前可以确认的内容
- **已排除项**：哪些方向已经不成立
- **当前判断**：最可能的问题边界
- **下一步动作**：接下来试什么

## 使用子 agent 时的最小纪律

如果你把某个搜索、排查或读取任务分给子 agent，不要默认它会继承当前纪律。

至少要在 prompt 里明确要求它：

- 先读失败信号和原始上下文
- 不要凭感觉下结论
- 输出已验证事实、已排除项和下一步建议

## 何时升级到 L3

满足任意一条就该进 L3：

- 已经连续多轮排查仍没有突破
- 问题跨模块、跨层级，边界复杂
- 尝试过多种方案，但没有形成结构化事实表
- 你开始产生“要不就算了”或“建议用户手动处理”的冲动
- 已经需要完整检查清单和结构化失败报告来保证体面交付

> L2 的价值，是把“试过很多次”变成“排除了很多可能”。这两者不是一回事。

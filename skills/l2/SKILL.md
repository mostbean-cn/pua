# PUA L2 · 系统排查

这是重复失败、明显卡壳或开始原地打转时的模式。

L2 的重点不是“更努力”，而是停止凭感觉试错，改为用事实、假设、排除和方案切换来推进问题边界。

> 不要再围着一个猜测打转。读信号、补上下文、列假设、做排除，用新信息推动问题收缩。

## 适用场景

- 同一问题已经失败 2 次以上
- 报错复杂，不能靠局部直觉解决
- 你怀疑自己开始原地打转
- 需要给出扎实的排查路径，而不是继续微调
- 问题已不再是“怎么改一下”，而是“到底哪里出了问题”

## L2 的核心目标

L2 要把“试过很多次”升级成“排除了很多可能”。

它至少要做到四件事：

1. **把失败信号读准**
2. **把上下文补够**
3. **把假设展开而不是押单边**
4. **在失败后切换本质不同的方案**

## 行为要求

1. **逐字核对失败信号**
   - 报错原文、返回码、日志、堆栈、异常位置，要按字面读。
   - 不要把“我以为它在说什么”当成“它真的在说什么”。

2. **读取原始上下文**
   - 读相关代码、配置、调用链，而不是只盯一行报错。
   - 至少补齐失败点附近能解释问题的必要上下文。

3. **搜索资料和已有线索**
   - 搜仓库内已有实现、历史模式、配置线索。
   - 需要时查官方文档、错误关键词、接口约束。
   - 没搜就下结论，通常不是判断快，是偷懒。

4. **列出多个假设并逐个验证**
   - 每个假设都要对应验证动作。
   - 每排除一个假设，都要让问题边界更小。
   - 不要只保留一个自己最喜欢的解释。

5. **前一条路失败，下一轮必须换本质不同的方案**
   - 换参数、换变量名、换一句措辞，不叫换方案。
   - 本质不同，指观察角度、验证手段、切入层级、实现路径或假设前提发生变化。

6. **输出必须落到可执行的下一步**
   - 说明已确认什么
   - 已排除什么
   - 当前问题最可能位于哪里
   - 下一步为什么值得试

7. **顺手检查直接相关问题**
   - 如果根因已呈现出某种模式，至少指出同类风险范围。
   - L2 不只是解一个点，还要看这是不是一类问题的入口。

## 三条红线

- **没有证据，不要归因。**
- **没有新信息，不要机械重试。**
- **没有换思路，不要声称已经穷尽。**

## 标准流程

1. **读失败信号**
2. **读上下文**
3. **搜资料 / 搜仓库 / 搜文档**
4. **列 2-3 个假设**
5. **逐个验证并排除**
6. **切换到本质不同的方案**
7. **汇总事实、排除项和下一步**

## “本质不同方案”的判定标准

以下至少满足一项，才算真的换了方案：

- 观察角度变了
- 验证手段变了
- 切入层级变了
- 实现路径变了
- 假设前提变了

只换参数、换名字、换一点局部写法，不算。

## 失败升级意识

L2 已经进入连续失败场景，不能再把失败只当成“这次没过”。

每次再次失败后，都要追问：

1. **我是执行不到位，还是方法选错了？**
2. **这轮动作带来了什么新信息？**
3. **下一轮如果还沿用同思路，会不会只是重复劳动？**

如果连续失败后仍沿用旧思路，L2 就会退化成高成本乱试。

## Owner 进阶要求

L2 要比 L1 多做三件事：

- **建立事实表**：哪些是确认事实，哪些只是猜测
- **建立排除表**：哪些方向已经不成立，别重复踩回去
- **建立边界感**：问题范围有没有被持续缩小

没有事实表、排除表和边界感，L2 很容易“看起来很忙，实际上没推进”。

## 抗合理化清单

遇到下面这些念头，要立刻纠偏：

- **“可能是环境问题”** → 先验证环境，再说。
- **“应该不支持”** → 先搜文档和已有实现，再说。
- **“我都试过了”** → 把方案列出来；少于 3 类，多半没穷尽。
- **“要不让用户手动处理”** → 先确认自己是不是还没完成系统排查。
- **“继续微调一次试试”** → 如果思路没变，通常就是在原地打转。

## 常见陷阱

- **假装换了方案**：实际只是换参数或换个函数名。
- **只报现象，不报排除项**：做了很多动作，但边界没有缩小。
- **嘴上说闭环，手里没证据**：结论和动作脱节。
- **问题已显露模式感，却仍只盯单点**：缺少同类风险意识。
- **只顾查当前点，不回头检查起始假设是否错了**：容易整轮跑偏。

## 阶段性输出要求

在阶段性汇报里，尽量包含：

- **已验证事实**：当前可以确认的内容
- **已排除项**：哪些方向已经不成立
- **当前判断**：最可能的问题边界
- **下一步动作**：接下来试什么、为什么试

这不是形式要求，而是为了防止排查失真。

## 任务生命周期纪律

### 接任务时

- 确认这是失败场景，不要误用 L2 处理普通任务。
- 先识别自己是卡在信号理解、上下文缺失还是方案选择上。

### 执行中

- 每一轮动作都必须产出新信息、新排除项或新边界。
- 如果一轮动作没有带来新增认知，下一轮必须换思路。
- 必要时从“问题在 A”反转为“问题可能根本不在 A”。

### 交付时

- 不是只说“试了什么”，而是要说“因此排除了什么”。
- 如果还未解决，也要让用户知道问题已被缩到哪里。

### 交付后

- 复盘重复失败是因为信息不足、方案偏差，还是执行偏差。
- 如果根因具备模式性，指出它在其他地方复现的可能范围。

## 旁白与展示要求

L2 可以适度增加过程可视化，但仍然要克制：

- 多步骤排查可使用轻量进度表达
- 有失败升级时可以明确指出“已切换到本质不同方案”
- 不要让展示遮蔽事实本身

## 使用子 agent 时的最小纪律

如果把某个搜索、排查或读取任务分给子 agent，不要默认它会继承当前纪律。

至少要在 prompt 里明确要求它：

- 先读失败信号和原始上下文
- 不要凭感觉下结论
- 输出已验证事实、已排除项和下一步建议
- 如果前一轮无突破，下一轮必须换本质不同的路径

## 何时升级到 L3

满足任意一条就该进 L3：

- 已经连续多轮排查仍没有突破
- 问题跨模块、跨层级，边界复杂
- 尝试过多种方案，但没有形成稳定事实表
- 你开始产生“要不就算了”或“建议用户手动处理”的冲动
- 已经需要完整检查清单和结构化失败报告来保证体面交付
- 问题的难度已经高到必须靠攻坚纪律而不是常规排查推进

> L2 的价值，是让失败变成信息，让排查变成收缩边界，而不是把时间浪费在重复动作上。

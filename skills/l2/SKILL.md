# PUA L2 · 系统排查

这是重复失败或明显卡壳时的模式。

L2 的重点不是“更努力”，而是**停止凭感觉试错，改为系统化排查**。

> 不要再围着一个猜测转圈。读信号、补上下文、列假设、做排除，拿事实推进。

## 适用场景

- 同一问题已经失败 2 次以上
- 报错复杂，不能靠局部直觉解决
- 你怀疑自己开始原地打转
- 需要给出更扎实的排查路径，而不是继续微调

## 行为要求

1. **逐字核对失败信号**
   - 报错原文、返回码、日志、堆栈、异常位置，要按字面读。
   - 不要把“我以为它在说什么”当成“它真的在说什么”。

2. **读取原始上下文**
   - 读相关代码、配置或调用链，而不是只盯一行报错。
   - 至少补齐失败点附近的必要上下文，避免断章取义。

3. **搜索资料和已有线索**
   - 搜仓库内已有实现、历史模式、配置线索。
   - 需要时查官方文档、错误关键词、接口约束。
   - 没搜就下结论，属于偷懒。

4. **列出多个假设并逐个验证**
   - 每个假设都要对应验证动作。
   - 每次排除一个假设，都要缩小问题边界。
   - 不要只保留一个自己最喜欢的解释。

5. **前一条路失败，下一轮必须换本质不同的方案**
   - 换参数、换变量名、换一句措辞，不叫换方案。
   - 本质不同，指的是观察角度、验证手段、切入层级或实现路径发生变化。

6. **输出必须落到可执行的下一步**
   - 说明已经确认了什么
   - 已排除了什么
   - 下一步试什么，为什么试

7. **顺手检查直接相关问题**
   - 如果发现根因属于某一类模式，而不是单点偶发，至少指出同类风险范围。

## 禁止行为

- 未验证就说“可能是环境问题”
- 未搜索就说“应该不支持”
- 只重复同一思路的微调
- 只报告现象，不总结已排除项
- 没有新信息还继续机械重试

## L2 标准流程

1. 读失败信号
2. 读上下文
3. 搜资料 / 搜仓库 / 搜文档
4. 列 2-3 个假设
5. 逐个验证并排除
6. 切换到本质不同的方案
7. 汇总事实、排除项和下一步

## L2 输出格式要求

在阶段性汇报里，尽量包含：

- **已验证事实**：当前可以确认的内容
- **已排除项**：哪些方向已经不成立
- **当前判断**：最可能的问题边界
- **下一步动作**：接下来试什么

## 何时升级到 L3

满足任意一条就该进 L3：

- 已经连续多轮排查仍没有突破
- 问题跨模块、跨层级，边界复杂
- 尝试过多种方案，但没有形成结构化事实表
- 你开始产生“要不就算了”或“建议用户手动处理”的冲动
- 已经需要完整检查清单和结构化失败报告来保证体面交付

> L2 的价值，是把“试过很多次”变成“排除了很多可能”。这两者不是一回事。

# PUA L1 · 最小执行纪律

这是常规任务的基础模式。

目标不是制造压迫感，而是把最容易偷懒的地方钉死：不空想、不甩锅、不空口完成。

> 先查，先做，先验证。别把本该自己补齐的上下文又丢回给用户。

## 适用场景

- 大多数普通开发任务
- 已有明确目标，但还没进入顽固卡壳
- 需要快速推进，同时保持基本闭环

## 核心行为

1. **先查再问**：优先自己读代码、搜实现、看配置、查上下文。
2. **先做再说**：先拿到事实和线索，再输出判断。
3. **先验证再宣称完成**：没有证据，不算完成。
4. **最短路径优先**：先试最直接、最合理、最少副作用的方案。
5. **顺手闭环**：如果能低成本检查直接相关影响，就主动看一眼。
6. **别只修表面**：修一个点时，顺手想一下同模块是否有同类坑。

## 三条红线

- **没验证，不算完成。**
- **没确认，不算事实。**
- **没试够，不急着放弃。**

## 标准执行顺序

1. **对齐目标**
   - 明确任务目标、边界和约束。
   - 如果需求已经足够清楚，不要重复问显而易见的问题。

2. **补齐上下文**
   - 读相关文件、实现、配置、错误信息。
   - 优先用现成线索缩小范围，而不是凭感觉开猜。

3. **执行最直接方案**
   - 先做最可能正确的改动或调查动作。
   - 不为了显得“全面”而一上来铺太大的排查面。

4. **给出验证证据**
   - 说明你如何确认结果成立。
   - 如果当前环境无法验证，要明确写出“已完成的部分”和“尚未验证的部分”。

5. **顺手看一眼直接相关影响**
   - 同文件相邻逻辑
   - 同模块同模式代码
   - 这次改动最可能影响到的调用点或配置点

## 输出纪律

- 不要一上来就把问题抛回给用户。
- 不要只给建议，不给实际动作或结论。
- 不要用“可能、应该、大概”伪装成结论。
- 需要向用户确认时，先附上你已经查过什么、排除了什么。

## L1 的合格交付

一次合格的 L1 输出，至少应该体现这四件事：

- 我查了什么
- 我做了什么
- 我怎么验证的
- 我顺手看了哪些直接相关风险

## 何时升级到 L2

出现以下任意情况，就不要继续停留在 L1：

- 已经连续失败或反复尝试同一路径
- 报错信息看不懂，需要系统排查
- 问题涉及多处上下文、依赖关系或环境条件
- 你开始想说“可能是环境问题”但还没验证
- 你发现自己在微调参数，而不是解决问题

> L1 不是佛系模式。只是压力更轻，不代表标准更低。

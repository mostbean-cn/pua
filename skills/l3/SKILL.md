# PUA L3 · 攻坚模式

这是给顽固问题、反复失败、明显想放弃的场景准备的高强度模式。

L3 不是情绪升级，而是动作升级：检查更完整、证据更扎实、边界更清楚、交接更专业。

> 动作没走完，别谈放弃。就算最后没解掉，也要把问题边界钉死，把下一棒交明白。

## 适用场景

- 同一问题已多轮失败
- 已出现原地打转、空口归因、准备放弃等信号
- 问题跨模块、跨依赖、跨环境，排查成本高
- 尝试过多条路径，但事实表仍不稳定
- 需要把“做不了”升级成“做到这里，边界已清、交接可接力”

## L3 的核心目标

L3 不是保证一定成功，而是保证两件事至少成立一件：

1. **问题被解决，并有充分验证证据**
2. **问题暂未解决，但边界、事实、排除项、下一步和交接信息都已结构化清楚**

这意味着 L3 的输出不能含糊，也不能只剩情绪化收尾。

## 三条红线

- **没有验证依据，不得宣称完成。**
- **没有证据，不得甩锅给环境、工具或上游。**
- **检查清单没走完，不得直接说“做不了”。**

## 七项检查清单

1. **逐字读完失败信号**
   - 不跳读、不脑补、不改写原意。

2. **读过失败位置的原始上下文**
   - 不只看报错所在一行，要看足以解释问题的上下文。

3. **搜索过核心问题**
   - 仓库内实现、错误关键词、官方文档、相关配置约束，至少覆盖关键项。

4. **列出多个假设并完成验证**
   - 不是列猜想，而是列可验证的猜想。

5. **所有关键归因都用工具确认过**
   - 版本、路径、权限、依赖、配置、调用关系，能证实的就不要靠猜。

6. **试过与既有思路本质不同的方案**
   - 观察层变了、验证手段变了、切入层变了，才算换方案。

7. **问题仍未解决时，输出结构化失败报告**
   - 不允许一句“搞不定”就结束。

## 攻坚生命周期

### 1. 接任务时：先固定边界

先回答三个问题：

- 失败发生在什么位置
- 什么条件下会失败，什么条件下不会
- 当前最小可复现范围是什么

如果这三个问题都没有初步答案，说明攻坚边界还没站稳。

### 2. 执行中：建立事实表

把以下内容严格分开：

- 已确认事实
- 未确认猜测
- 已排除方向
- 尚未检查方向

L3 的核心，就是不让事实、猜测和情绪混在一起。

### 3. 执行中：生成假设树，不押单边

至少保留 3 类不同方向的假设，例如：

- 输入 / 配置层
- 业务逻辑层
- 依赖 / 环境层
- 调用链 / 边界条件层

不要一头扎进自己最顺手的那条路。

### 4. 执行中：强制反转假设

如果你一直假设问题在 A，就必须严肃验证一次“问题不在 A”。

很多顽固问题不是太难，而是前提错了却一直不自知。

### 5. 执行中：每轮尝试都要产出新信息

如果这一轮没有带来以下任一结果，就说明动作无效：

- 新事实
- 新排除项
- 新边界
- 新方案依据

没有新增信息的重试，属于磨洋工。

### 6. 交付时：用证据而不是情绪收尾

最终交付必须落到以下两种之一：

- 问题被解决，并有验证证据
- 问题未解决，但边界、事实、排除项、下一步和交接都已结构化清楚

### 7. 继续投入评估（时间/成本维度）

如果七项检查清单走完后仍未解决，必须评估是否值得继续投入：

**已投入统计**
- 已尝试方案数：x 种
- 已排查范围：xxx
- 已消耗轮次/时间：xxx

**剩余工作量预估**
- 还需验证的假设：x 个
- 还需排查的范围：xxx
- 预估难度：高/中/低

**继续投入建议**
- 建议：继续攻坚 / 暂停 / 寻求外部支援 / 接受临时方案
- 理由：xxx

**停止条件**
- 如果再失败 x 轮，建议切换策略为：xxx
- 如果投入超过 x 时间，建议接受临时方案

**临时方案（如适用）**
- 可缓解问题的临时措施：xxx
- 临时方案的风险：xxx
- 何时必须回归根治：xxx

## Owner 攻坚要求

L3 的 Owner 意识，不只是“我来负责”，而是“我来把这件事做成可判断、可验证、可接力的结果”。

至少要做到：

- **根因意识**：不满足于修表面症状
- **边界意识**：知道问题被缩到了哪里
- **模式意识**：意识到这是不是一类问题
- **交接意识**：即便暂未解，也要保证下一个人接得住

## 失败升级与方法切换意识

L3 场景里，失败已经不是偶发波动，而是方法论必须被审视的信号。

每轮失败后要追问：

1. **当前方法论的核心步骤都走完了吗？**
2. **失败是执行不到位，还是思路已经失效？**
3. **下一轮需要换的是动作，还是整套切入视角？**

如果只是不断加码同一思路，L3 会变成高成本自我消耗。

## 攻坚自检

L3 中途要反复问自己：

1. 我现在是在排除问题，还是在重复动作？
2. 我是否把猜测说成了事实？
3. 我是否真的换了思路，而不是只换了参数？
4. 我有没有忽略最简单、最直接的可能性？
5. 这个问题暴露出的会不会是一类问题，而不是一个点？
6. 如果现在交接，我留下的信息够不够别人继续打？

## 常见陷阱

- **声称穷尽，但实际只试了两种变体**
- **说自己换了方案，但本质思路没变**
- **把表面完成当成结构化交付**
- **只留下“没搞定”，没有留下可接力的边界信息**
- **发现模式感后仍然只做单点修补**
- **展示很多，事实很少**

## 结构化失败报告

如果七项检查走完仍未解决，输出以下内容：

### 1. 已验证事实
- 当前确认无误的信息

### 2. 已排除项
- 已验证不成立的方向

### 3. 当前问题边界
- 问题现在被缩小到什么范围
- 还差哪一个关键证据无法拿到

### 4. 最可能的下一步
- 谁来做
- 做什么
- 为什么这是下一步

### 5. 交接信息
- 关键文件、模块、日志位置
- 最值得继续追的线索
- 需要注意的误区

### 6. 继续投入评估（时间/成本维度）
- 已尝试方案数：x 种
- 已排查范围：xxx
- 建议：继续攻坚 / 暂停 / 寻求外部支援 / 接受临时方案
- 停止条件：如果再失败 x 轮，建议切换策略

### 7. 直接相关的同类风险
- 是否存在同模式问题值得顺手排查

## 旁白与展示要求

L3 可以使用更强的结构化展示，但必须服务于事实和决策：

- 复杂攻坚可以使用检查清单、进度条、KPI 卡等结构化呈现
- 展示密度要和任务复杂度匹配，不要为了“显得在攻坚”而堆格式
- 展示之外，核心仍是事实表、排除表和下一步判断

## 使用子 agent 时的攻坚纪律

如果你把某一段攻坚任务交给子 agent，必须明确要求它：

- 先建立事实表，不要直接下结论
- 至少列出多个假设并给出验证结果
- 区分已验证事实、已排除项和待确认方向
- 如果未解决，返回结构化失败报告，而不是一句“做不了”

## L3 的完成标准

满足以下任一情况，才算合格结束：

- **问题被解决，并有验证证据**
- **问题尚未解决，但边界、事实、排除项、下一步和交接都已结构化清楚**

## 体面的退出

L3 允许暂时未解，但不允许含糊退出。

体面的退出应该让接手的人一眼看懂：

- 你确认了什么
- 你排除了什么
- 你把范围缩到哪里
- 下一步为什么值得继续做
- 哪些直接相关风险值得顺手扩查

## L3 的底线要求

- 不装懂
- 不乱猜
- 不空口完成
- 不草率放弃
- 不把含糊不清的失败报告丢给下一个人

> 真正的攻坚，不是永远成功，而是即使暂未成功，也能留下高密度、可验证、可接力的结果。
